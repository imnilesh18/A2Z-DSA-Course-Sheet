## 📝 Problem Statement

You are given an integer array `nums`.

A **XOR triplet** is defined as the XOR of three elements: `nums[i] XOR nums[j] XOR nums[k]` where `i <= j <= k`.

Return the number of **unique** XOR triplet values from all possible triplets `(i, j, k)`.

---

## 🚀 Constraints

*   $1 \le \text{nums.length} \le 1500$
*   $1 \le \text{nums}[i] \le 1500$

## 🧠 Approach & Explanation

Unlike Part I of this problem where elements are strictly permutations from `1` to `N`, Part II features random numbers with a length constraint of $N \le 1500$. This specific constraint is a massive hint: an $O(N^2)$ solution will pass, but $O(N^3)$ will result in a Time Limit Exceeded (TLE) error. 

<details>
<summary><b>Click to reveal the detailed walkthrough</b></summary>

1. **Precomputing Pairs:** 
   Instead of looping three times to form `(i, j, k)`, we can calculate the XOR of all possible valid pairs `(i, j)` first. 
   Since the XOR operation is commutative and associative, the specific order of the elements doesn't matter (e.g., $A \oplus C \oplus D$ is exactly the same as $A \oplus D \oplus C$). Therefore, we don't have to strictly track the `i <= j <= k` index order when generating the final triplet values.

2. **Bounding the Search Space:** 
   A crucial property of the XOR operation is that it never generates a "new" higher bit that wasn't already present or possible within the operands.
   If we find the maximum element in our array, the result of any XOR operations involving elements of this array will **never** exceed the next power of 2 greater than this maximum element. 
   For example, if the max element is `9` (binary `1001`), the next power of 2 is `16` (`10000`). Our XOR results will strictly stay below `16`.

3. **Optimization Leap:** 
   We can first store all pair XORs in an unordered set. Then, we iterate through these unique pair values and XOR them against every element in `nums` to find the triplet values. 
   However, hash sets carry an internal hashing overhead. Just like optimizing a deployment pipeline to strip away unnecessary latency, or refining a Formula 1 car's aerodynamics to shave milliseconds off a lap time, we can replace the hash set with a fixed-size Boolean vector. Since we know the absolute upper bound (the next power of 2), a boolean array mapped directly to these values offers lightning-fast $O(1)$ lookups without hash collisions.

</details>

## 💻 Solution Code

### Brute-force Solution (Set Approach)

```cpp
// Intuition: Instead of an O(N^3) loop, precompute all possible pair XORs and store them. Then, XOR these unique pair values with all array elements to form the final triplets.
// Approach: Generate all pair XORs and store them in a hash set to guarantee uniqueness. Iterate through this set and XOR each value with every element in nums, inserting the results into a second set. The size of the second set is the answer.
// Time Complexity: O(N^2 + N * maxEl), where maxEl is the maximum element in nums. Finding pairs takes O(N^2), and iterating the set takes at most O(maxEl).
// Space Complexity: O(maxEl), because the maximum possible XOR value is strictly less than the next power of 2 after the maximum element.

class Solution {
public:
    int uniqueXorTriplets(vector<int>& nums) {
        int n = nums.size();

        // Store all Pair Xors to ensure uniqueness
        unordered_set<int> s1;
        for(int i = 0; i < n; i++) {
            for(int j = i; j < n; j++) {
                s1.insert(nums[i] ^ nums[j]);
            }
        }

        // Find triplet xors by combining pair xors with remaining elements
        unordered_set<int> s2;
        for(int pairXor : s1) {
            for(int &num : nums) {
                s2.insert(pairXor ^ num);
            }
        }

        // Return the count of unique triplet values
        return s2.size();
    }
};

/*
*
* Dry Run
* Nums = [1, 3]
* 1. Pairs: (1^1)=0, (1^3)=2, (3^3)=0 -> s1 = {0, 2}
* 2. Triplets:
*    Using 0: 0^1=1, 0^3=3
*    Using 2: 2^1=3, 2^3=1
* 3. s2 = {1, 3}
* 4. Return size of s2 -> 2
*
*/

```

### Optimized Solution (Vector Approach)

```cpp
// Intuition: Hash sets introduce internal hashing and allocation overhead. Since the maximum possible XOR value is mathematically bounded by the next power of 2 after the array's maximum element, a fixed-size boolean vector provides much faster direct-access mapping.
// Approach: Find the array's max element and calculate the next power of 2 (T). Create two boolean vectors of size T. Map all pair XORs by setting their indices to true in the first vector. Then, iterate up to T, and for every true index, XOR it with all array elements, marking the results in the second vector.
// Time Complexity: O(N^2 + N * maxEl), where maxEl is the maximum element. This eliminates the hidden constants and overhead of unordered_set hashing.
// Space Complexity: O(maxEl), strictly bounded by the next power of 2 after the maximum element in nums.

class Solution {
public:
    int uniqueXorTriplets(vector<int>& nums) {
        int n = nums.size();
        
        // Find the maximum element to determine the upper bound
        int maxEl = *max_element(begin(nums), end(nums));

        // Calculate the next power of 2 greater than maxEl
        int T = 1;
        while(T <= maxEl) {
            T <<= 1; // T = T * 2
        }

        // Vectors act as direct-mapped frequency/presence arrays
        vector<bool> s1(T, false); // XOR pair values set to true
        vector<bool> s2(T, false); // XOR triplet values set to true

        // Store presence of all pairwise XORs
        for(int i = 0; i < n; i++) {
            for(int j = i; j < n; j++) {
                s1[nums[i] ^ nums[j]] = true;
            }
        }

        // Generate triplet XORs using existing pairs
        for(int i = 0; i < T; i++) {
            if(s1[i] == true) {
                for(int &num : nums) {
                    s2[i ^ num] = true;
                }
            }
        }

        // Count unique triplet values
        int count = 0; 
        for(int i = 0; i < T; i++) {
            if(s2[i] == true) {
                count++;
            }
        }
        
        return count;
    }
};

/*
*
* Dry Run
* Nums = [1, 3], Max = 3
* Next power of 2 (T) = 4
* s1 = [F, F, F, F], s2 = [F, F, F, F]
* Pairs: (1^1)=0, (1^3)=2, (3^3)=0. 
* s1 becomes [T, F, T, F] (Indices 0 and 2 are true)
* Triplets checking s1:
* i=0 is True -> 0^1=1 (s2[1]=T), 0^3=3 (s2[3]=T)
* i=2 is True -> 2^1=3 (s2[3]=T), 2^3=1 (s2[1]=T)
* s2 becomes [F, T, F, T]
* Count of true values in s2 = 2
*
*/

```

## 🔑 Key Insights

* **Constraint Analysis:** Identifying that $N \le 1500$ instantly rules out $O(N^3)$ and strongly suggests precomputing components in $O(N^2)$.
* **Bitwise Bounds:** The realization that an XOR operation cannot mathematically produce a value larger than the next power of 2 of the maximum operand is the cornerstone of the optimized approach.
* **Data Structure Choice:** Upgrading from an `unordered_set` to a `vector<bool>` drastically reduces system memory overhead and prevents hashing collisions, turning a good solution into a highly robust, competitive one.
